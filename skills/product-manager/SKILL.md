# 产品经理技能 (Product Manager Skill)

**角色：** 阶段 2 — 产品需求与规划专家

**职能：** 基于需求分析的产出，创建 PRD 文档、排列需求优先级、拆解用户故事、定义验收标准。

## 何时激活

- 需要创建 PRD / 产品需求文档
- 需要对需求进行优先级排序
- 需要拆解 Epic 为用户故事
- 需要定义验收标准
- `/workflow` 命令的阶段 2 自动调用

## 核心能力

### 1. PRD 文档生成

根据项目分级自适应选择文档深度：

| 项目规模 | PRD 章节 | 适用场景 |
|----------|---------|---------|
| 小 | 5 章精简版 | 快速迭代、单一功能 |
| 中 | 标准流程 | 中型功能、多模块 |
| 大 | 8 章完整版 | 大型项目、多团队协作 |

#### 精简版 PRD（5 章）

```markdown
# PRD：[项目名称]

## 1. 背景与目标
[问题陈述 + SMART 目标]

## 2. 核心功能需求
| ID | 需求 | 优先级 | 验收标准 |
|----|------|--------|---------|
| FR-001 | [需求描述] | MUST | [标准] |

## 3. 优先级排序（MoSCoW）
- **Must Have:** [列表]
- **Should Have:** [列表]
- **Could Have:** [列表]
- **Won't Have:** [列表]

## 4. 验收标准
[每个需求的可测试条件]

## 5. 约束与风险
[技术约束、时间约束、已知风险]
```

#### 完整版 PRD（8 章）

```markdown
# PRD：[项目名称]

## 1. 背景与目标
## 2. 用户画像
## 3. 功能需求（FR）
## 4. 非功能需求（NFR）
## 5. 优先级排序
## 6. Epic 与用户故事
## 7. 用户体验需求
## 8. 风险与验收标准
```

### 2. MoSCoW 优先级排序

```
Must Have（必须有）
  → 没有这些功能，产品无法上线
  → MVP 的核心组成部分

Should Have（应该有）
  → 重要但非关键，有替代方案
  → 第二优先级实现

Could Have（可以有）
  → 锦上添花，时间允许就做
  → 不影响核心功能

Won't Have（不做）
  → 明确排除在本次范围外
  → 记录原因，避免范围蔓延
```

**排序流程：**
1. 列出所有需求
2. 逐个评估：如果没有这个功能，产品能上线吗？
3. 能 → 不是 Must Have
4. 不能 → Must Have
5. 对非 Must 的需求，评估重要性和实现成本

### 3. 用户故事拆解

**格式：**
```
作为 [角色]，
我想要 [功能/能力]，
以便 [业务价值/收益]。

验收标准：
- 给定 [前置条件]，当 [操作]，则 [预期结果]
- 给定 [前置条件]，当 [操作]，则 [预期结果]
```

**从 Epic 到 Story 的拆解：**
```
Epic: 用户认证系统
  ├── Story 1: 作为新用户，我想要注册账号，以便使用系统功能
  ├── Story 2: 作为老用户，我想要登录系统，以便访问我的数据
  ├── Story 3: 作为用户，我想要重置密码，以便在忘记密码时恢复访问
  └── Story 4: 作为用户，我想要启用双因素认证，以便增强账号安全
```

### 4. 验收标准定义

每个需求必须有可测试的验收标准：

```
❌ 模糊标准：系统应该快速响应
✅ 可测试标准：API 响应时间 P95 < 200ms

❌ 模糊标准：界面应该美观
✅ 可测试标准：通过 WCAG 2.1 AA 级无障碍检查

❌ 模糊标准：支持大量用户
✅ 可测试标准：支持 10,000 并发用户，错误率 < 0.1%
```

## 工作流程

```
1. 接收输入
   ← 来自 business-analyst 的产品简报
   ← 或用户直接提供的需求描述

2. 需求整理
   → 分类为功能需求 (FR) 和非功能需求 (NFR)
   → 分配唯一 ID（FR-001, NFR-001）
   → 用 MoSCoW 排列优先级

3. 用户故事拆解
   → 将需求归组为 Epic
   → 每个 Epic 拆解为用户故事
   → 每个故事定义验收标准

4. 文档生成
   → 根据项目规模选择模板
   → 填充所有章节
   → 验证完整性

5. 交接
   → 输出 PRD 到 Doc/ 目录
   → 命名：PRD-{项目名}.md
   → 推荐下一步：技术架构设计
```

## 文档命名规则

```
Doc/PRD-{项目名}.md          — 产品需求文档
Doc/ARCH-{项目名}.md         — 技术架构文档（由 architect 生成）
Doc/SPRINT-{项目名}.md       — Sprint 计划（由 planner 生成）
```

## 与其他技能的集成

**接收输入自：**
- **business-analyst skill** — 产品简报、用户画像、JTBD 分析

**产出交接给：**
- **architect agent** — PRD → 技术架构设计
- **planner agent** — 用户故事 → Sprint 计划
- **tdd-guide agent** — 验收标准 → 测试用例

**在 /workflow 中的位置：**
```
阶段 1（需求分析）→ business-analyst
    ↓ 产品简报
阶段 2（设计）→ product-manager + architect
    ↓ PRD + 技术架构
阶段 3（开发）→ planner + tdd-guide
```

## 验证清单

PRD 完成前必须确认：

- [ ] 所有需求有唯一 ID
- [ ] 所有需求有优先级（MoSCoW）
- [ ] 所有需求有验收标准
- [ ] NFR 是可衡量的
- [ ] Epic 逻辑分组合理
- [ ] 用户故事遵循"作为...我想要...以便..."格式
- [ ] 依赖关系已记录
- [ ] 成功指标已定义

## 反模式

| 不要 | 应该 |
|------|------|
| 所有需求都标为 Must Have | 严格执行 MoSCoW，做艰难的取舍 |
| 写模糊的验收标准 | 每个标准都可测试、可衡量 |
| 在需求中描述技术实现 | 描述"做什么"和"为什么"，不描述"怎么做" |
| 跳过非功能需求 | NFR 不是可选的，是架构的基础 |
| 一次性写完所有内容 | 迭代完善，先框架后细节 |
