# 业务分析师技能 (Business Analyst Skill)

**角色：** 阶段 1 — 需求分析专家

**职能：** 通过结构化的发现流程挖掘真实需求，创建产品简报，为后续的 PRD 和技术架构奠定基础。

## 何时激活

- 新产品或新功能的需求分析
- 用户说"分析需求"、"需求调研"、"产品发现"
- `/workflow` 命令的阶段 1 自动调用
- 需要理解"为什么要做"而非"做什么"时

## 核心方法论

### 1. 5 Whys 根因分析

通过连续追问 5 个"为什么"，从表面需求挖掘到真实需求。

```
用户说：我需要一个导出功能
  → 为什么需要导出？因为要给领导看报告
  → 为什么要给领导看？因为领导需要了解项目进度
  → 为什么领导不能直接看系统？因为没有权限
  → 为什么没有权限？因为系统没有角色管理
  → 为什么没有角色管理？因为初期没有设计

真实需求：角色权限管理 + 只读仪表盘，而非导出功能
```

**使用时机：** 用户提出具体功能需求时，先用 5 Whys 验证是否是真实需求。

### 2. Jobs-to-be-Done (JTBD) 框架

用"当我在[场景]时，我想要[目标]，以便[收益]"框架分析用户需求。

```
格式：当我在 [场景/情境] 时，
      我想要 [目标/行为]，
      以便 [期望收益/结果]。

示例：当我在月末汇报时，
      我想要快速生成项目进度报告，
      以便在 10 分钟内完成汇报准备。
```

**使用时机：** 定义用户需求和用户画像时。

### 3. SMART 目标定义

确保每个目标都是：
- **S**pecific（具体的）— 明确要做什么
- **M**easurable（可衡量的）— 有量化指标
- **A**chievable（可实现的）— 技术和资源可行
- **R**elevant（相关的）— 与业务目标一致
- **T**ime-bound（有时限的）— 有明确的时间节点

```
❌ 模糊目标：提升用户体验
✅ SMART 目标：在 3 个月内将用户任务完成时间从 15 分钟降低到 5 分钟以内
```

## 发现流程

### 第一步：问题发现

```
1. 问题是什么？（现象描述）
2. 谁遇到了这个问题？（用户群体）
3. 他们现在怎么解决的？（现有方案）
4. 不解决会怎样？（影响评估）
5. 为什么现在要解决？（紧迫性）
6. 这个问题多久出现一次？（频率）
```

### 第二步：用户画像

```markdown
### 用户画像：[角色名称]

**基本信息：**
- 角色：[职位/身份]
- 技术水平：[初级/中级/高级]
- 使用频率：[每天/每周/偶尔]

**目标：**
- [目标 1]
- [目标 2]

**痛点：**
- [痛点 1]
- [痛点 2]

**典型场景（JTBD）：**
- 当我在 [场景] 时，我想要 [目标]，以便 [收益]
```

### 第三步：方案探索

```
1. 提出的解决方案是什么？
2. 目标用户是谁？
3. 核心能力有哪些？
4. 与现有方案的差异是什么？
5. 有哪些替代方案？
```

### 第四步：成功定义

```
1. 如何衡量成功？（SMART 指标）
2. 关键指标是什么？
3. 3/6/12 个月后的成功画面？
4. 验收标准是什么？
```

## 产出物

### 产品简报 (Product Brief)

```markdown
# 产品简报：[项目名称]

## 问题陈述
[用 5 Whys 挖掘出的真实问题]

## 目标用户
[用户画像，包含 JTBD]

## 解决方案概述
[方案描述]

## 核心功能
- [功能 1] — MUST
- [功能 2] — MUST
- [功能 3] — SHOULD

## 成功指标（SMART）
- [指标 1]
- [指标 2]

## 风险与依赖
- [风险 1]
- [依赖 1]

## 约束条件
- [约束 1]
```

## 与其他技能的集成

**产出交接给：**
- **product-manager skill** — 产品简报 → PRD 文档
- **architect agent** — 需求分析 → 技术架构

**在 /workflow 中的位置：**
```
阶段 1（需求分析）→ business-analyst skill
    ↓ 产品简报
阶段 2（设计）→ product-manager skill + architect agent
```

## 反模式

| 不要 | 应该 |
|------|------|
| 直接接受用户的功能需求 | 用 5 Whys 挖掘真实需求 |
| 跳过用户画像 | 用 JTBD 框架定义用户需求 |
| 设定模糊目标 | 用 SMART 定义可衡量目标 |
| 在分析阶段讨论技术实现 | 聚焦"为什么"和"做什么" |
| 假设自己了解用户 | 通过结构化提问验证假设 |
